## 要旨

本文書は、異種コンピュートノードの分散ネットワーク上で並列 AI エージェント実行を調整するためのプロトコルである Mahalaxmi フェデレーション・オーケストレーション・プロトコル（MFOP）を定義します。MFOP は、ノードの識別と登録、ケイパビリティ広告、コンプライアンスゾーン対応ジョブルーティング、意味的入力分割、暗号署名付き請求レシート、設定可能な経済的精算、ならびに AI 安全ポリシー検証と実行サンドボックス分離を用いた階層型セキュリティモデルを規定します。

MFOP は、単一組織が所有・運用するノードで構成されるプライベートエンタープライズメッシュ、プラットフォームプロバイダーが運用するマネージドクラウドプール、および任意のノードオペレーターが経済的精算を対価としてコンピュートを提供できるオープンコミュニティマーケットプレイスという、3 つの同時展開構成で動作するよう設計されています。本プロトコルは基盤となる AI モデルプロバイダーに依存せず、AI 安全・コンプライアンスの状況の進化に合わせて発展するよう設計されています。

## 1. はじめに

エンタープライズ環境における大規模言語モデル（LLM）展開の拡大により、管轄区域および業界によって異なるコンプライアンス、請求、および安全要件を満たしながら、異種コンピュートインフラにまたがることができる調整レイヤーの必要性が生じています。

MFOP は、フェデレーテッド AI オーケストレーションのためのプロトコルを定義することで、この必要性に対応します。フェデレーションは、それぞれ異なるコンプライアンス体制のもとで異なる主体によって運用される可能性のある、1 つ以上のコンピュートノードで構成されます。サブミッター（ユーザー、アプリケーション、または自動化システム）はジョブをフェデレーションに提出します。フェデレーションは、ジョブのコンプライアンスゾーン要件、ノードのケイパビリティ広告、および適用される経済条件に基づいて、ジョブを適切なノードにルーティングします。

本仕様は、適合 MFOP フェデレーションのすべてのコンポーネントについて、ワイヤプロトコル、データフォーマット、暗号機構、および動作要件を定義します。

## 2. 用語

本文書における "MUST"、"MUST NOT"、"REQUIRED"、"SHALL"、"SHALL NOT"、"SHOULD"、"SHOULD NOT"、"RECOMMENDED"、"NOT RECOMMENDED"、"MAY"、および "OPTIONAL" というキーワードは、BCP 14 [RFC2119] [RFC8174] に記載されているとおりに解釈されるものとします。各キーワードの日本語 RFC 相当表現：「しなければならない (MUST)」「してはならない (MUST NOT)」「必須 (REQUIRED)」「するものとする (SHALL)」「すべきである (SHOULD)」「推奨される (RECOMMENDED)」「してもよい (MAY)」。

**フェデレーション** — 共通のガバナンス設定のもとで運用される、1 つ以上の MFOP 準拠コンピュートノードの論理的なグループ。

**ノード** — フェデレーションに登録され、AI ワークロードを受け付け、実行し、返却するコンピュートリソース。ノードは単一サーバー、クラスター、またはクラウドコンピュートプールであってよい。

**サブミッター** — AI ワークロードを実行のためにフェデレーションに提出するエンティティ（ユーザー、アプリケーション、または自動化システム）。

**コンプライアンスゾーン** — ジョブルーティング、データ処理、および出力検証を制約する名前付きポリシーコンテキスト。定義されたゾーン：public、enterprise (SOC2)、hipaa、sox、fedramp。

**ジョブ** — 実行のためにフェデレーションに提出される AI ワークロードの個別単位。ジョブはペイロード、コンプライアンスゾーンのアサーション、および請求承認を持つ。

**レシート** — 完了したジョブ実行の暗号署名付き記録。トークン数、タイムスタンプ、ノード識別情報、および請求額を含む。

**経済的精算** — 蓄積された請求レシートをサブミッター、ノードオペレーター、およびプラットフォーム間の資金移動に変換するプロセス。

**PAK Key（プラットフォーム API キー）** — フェデレーション API エンドポイントへのアクセスを承認するためにプラットフォームが発行するベアラー資格情報。

**NeMo Guardrails** — AI 安全ポリシー検証と出力フィルタリングのために MFOP ノードが使用する NVIDIA NeMo 安全フレームワーク。

## 3. ノード識別と登録

MFOP フェデレーション内の各ノードは、安定したグローバルに一意なノード識別子（node_id）によって識別されます。node_id は登録時に割り当てられる 128 ビットの UUID（バージョン 4）であり、ノードの再起動やソフトウェアアップグレードをまたいで持続します。

**3.1 登録フロー**

ノードは、フェデレーションの登録エンドポイント（POST /v1/federation/nodes/register）に NodeRegistrationRequest を送信することで登録を開始します。リクエストには以下を含めなければならない (MUST)：

- node_id：候補 UUID（フェデレーションはこれを上書きしてもよい (MAY)）
- operator_id：登録するオペレーターアカウントの UUID
- display_name：ノードの人間が読みやすい名前（最大 64 文字）
- public_key：レシート署名に使用する base64url エンコードの Ed25519 公開鍵
- capability_advertisement：CapabilityAdvertisement オブジェクト（セクション 4 参照）
- compliance_zones：ノードが処理の認定を受けているコンプライアンスゾーンのセット
- endpoint_url：ノードがジョブ提出を受け付ける HTTPS URL

フェデレーションは、割り当てられた node_id、後続の認証済み呼び出しのための registration_token、およびフェデレーションの現在の請求設定を含む NodeRegistrationResponse を返します。

**3.2 再登録と鍵のローテーション**

ノードは Ed25519 鍵ペアをローテーションする際に再登録しなければならない (MUST)。鍵のローテーション中、ノードは旧秘密鍵で署名した、旧公開鍵と新公開鍵の両方を含む再登録リクエストを提出します。フェデレーションは新しい鍵を受け入れる前に旧鍵署名を検証します。いずれかの鍵で署名されたレシートが受け入れられる 24 時間の重複ウィンドウが存在します。

**3.3 ノードのヘルスと登録解除**

ノードは少なくとも 60 秒ごとに POST /v1/federation/nodes/{id}/heartbeat にハートビートを送信しなければならない (MUST)。連続した 3 つのハートビートウィンドウを逃したノードは INACTIVE とマークされ、ルーティングから除外されます。ノードは DELETE /v1/federation/nodes/{id} を通じて自発的に登録解除することができます。

## 4. ケイパビリティ広告

ノードのケイパビリティ広告は、ノードで利用可能な AI モデル、ジョブルーティングに関連するハードウェア特性、およびノードオペレーターが保有するコンプライアンス認証を宣言します。

**4.1 CapabilityAdvertisement オブジェクト**

CapabilityAdvertisement オブジェクトには以下のフィールドが含まれます：

- models：ModelDescriptor オブジェクトの配列（4.2 参照）
- hardware_class：{ cpu、gpu_consumer、gpu_datacenter、tpu } のいずれか
- vram_gb：推論に使用可能な GPU VRAM の合計（ギガバイト単位、CPU ノードは 0）
- max_context_tokens：ノードが対応できる最大コンテキストウィンドウ
- max_concurrent_jobs：ノードが同時に実行するジョブの最大数
- compliance_certifications：認証識別子の配列（例：「soc2-type2」、「hipaa-baa」、「fedramp-moderate」）
- nemo_rails_version：ノードにインストールされている NeMo Guardrails ランタイムのバージョン

**4.2 ModelDescriptor**

ノードで利用可能な各モデルは ModelDescriptor によって記述されます：

- model_id：正規モデル識別子文字列（例：「meta-llama/Meta-Llama-3-70B-Instruct」）
- model_family：{ llama、mistral、gemma、falcon、phi、custom } のいずれか
- parameter_count_b：10 億パラメーター単位のおおよそのパラメーター数
- quantization：{ fp16、bf16、int8、int4、none } のいずれか
- context_window_tokens：このモデルの最大コンテキストウィンドウ
- supports_tool_use：ブール値
- supports_vision：ブール値

**4.3 ケイパビリティの更新**

ノードは、利用可能なモデルまたはハードウェア構成が変更された際に、PUT /v1/federation/nodes/{id}/capabilities を通じてケイパビリティ広告を更新しなければならない (MUST)。フェデレーションは更新されたケイパビリティ広告を 30 秒以内にルーティングレイヤーに伝播します。

## 5. コンプライアンスゾーン対応ジョブルーティング

MFOP は各ジョブをそのコンプライアンスゾーン要件を満たすノードにルーティングします。コンプライアンスゾーンの充足はハード制約です：ジョブのコンプライアンスゾーンの認定を受けていないノードにジョブをルーティングしてはならない (MUST NOT)。

**5.1 コンプライアンスゾーン**

MFOP は、制限の低いものから高いものへの順に 5 つのコンプライアンスゾーンを定義します：

- public：ベースラインの NeMo 安全レール以外のコンプライアンス要件なし。汎用 AI ワークロードに適しています。
- enterprise (SOC2)：SOC 2 Type II 認証が必要。データ所在地検出、API 資格情報漏洩検出、およびアクセスログ強制適用を追加します。
- hipaa：HIPAA BAA が必要。PHI パターン検出、PHI の匿名化、および最小必要出力チェックを追加します。
- sox：SOX コンプライアンス管理が必要。金融 PII 分離、価格予測ブロッキング、および MNPI 検出を追加します。
- fedramp：FedRAMP 認可が必要。CUI 処理、輸出規制検出、および機密区分マーキング強制適用を追加します。

**5.2 ルーティングアルゴリズム**

ジョブを受信すると、ルーティングレイヤーは以下のアルゴリズムを実行します：

1. フィルタリング：ジョブのコンプライアンスゾーンの認定を受けており、ステータスが ACTIVE のすべてのノードを識別する。
2. フィルタリング：max_context_tokens がジョブの推定トークン数より少ないノードを除外する。
3. フィルタリング：max_concurrent_jobs が現在いっぱいのノードを除外する。
4. スコアリング：残りの各ノードについてルーティングスコアを計算する：score = w_latency × latency_score + w_cost × cost_score + w_affinity × affinity_score。デフォルトの重み：w_latency = 0.4、w_cost = 0.4、w_affinity = 0.2。
5. 選択：最高スコアのノードにルーティングする。同点の場合は一様ランダムに選択する。

すべてのフィルターを満たすノードが存在しない場合、ジョブは設定可能なタイムアウト（デフォルト：120 秒）でキューに入れられます。タイムアウト内にノードが利用可能にならない場合、フェデレーションは Retry-After ヘッダー付きの HTTP 503 を返します。

**5.3 アフィニティルール**

サブミッターはジョブ提出時にアフィニティルールを指定してもよい (MAY)：

- node_affinity：優先するノード ID のリスト（ソフト優先）
- anti_affinity：除外するノード ID のリスト（ハード制約）
- geography：優先する地理的地域（ISO 3166-1 alpha-2 国コード）

アフィニティルールは affinity_score コンポーネントにのみ影響します。コンプライアンスゾーン認証と容量はハード制約のままです。

## 6. 意味的入力分割

入力が単一ノードの max_context_tokens を超えるジョブについて、MFOP は入力を一貫性のあるサブジョブに分割し、各サブジョブを独立してルーティングし、結果を集約する意味的分割メカニズムを提供します。

**6.1 分割戦略**

MFOP は 3 つの分割戦略を定義します：

- sliding_window：入力を設定可能なサイズと重複幅の重複ウィンドウに分割します。境界でのコンテキストの連続性が重要なタスクに適しています（例：長文書の要約）。
- semantic_boundary：検出された意味的境界（段落区切り、セクションヘッダー、トピック転換）で分割します。サブジョブのサイズが可変になるコストはあるものの、より一貫性の高いサブジョブを生成します。
- task_decomposition：入力を構造化タスクリストとして解釈し、各タスクを独立したサブジョブとしてルーティングします。入力が MFOP TaskList スキーマに準拠していることが必要です。

**6.2 分割リクエスト**

サブミッターはジョブ提出時に partition_strategy を設定することで分割実行を要求します。フェデレーションの分割エンジンが入力を分割し、サブジョブ ID（parent_job_id + シーケンス番号）を割り当て、各サブジョブを独立してルーティングします。サブジョブは親ジョブのコンプライアンスゾーンと請求承認を継承します。

**6.3 集約**

すべてのサブジョブが完了すると、フェデレーションの集約レイヤーがシーケンス番号順に結果を組み立てます。sliding_window 分割の場合、集約器は最長共通部分列マージを使用して重複領域のコンテンツを重複排除します。組み立てられた結果は、sub_job_receipts の配列を含む単一の JobResult としてサブミッターに返されます。

## 7. 暗号署名付き請求レシート

完了したすべてのジョブ実行は、実行ノードによって署名された BillingReceipt を生成します。署名済みレシートは経済的精算と紛争解決のための権威ある記録です。

**7.1 レシートの構造**

BillingReceipt には以下が含まれます：

- receipt_id：このレシートに固有の UUID（バージョン 4）
- job_id：完了したジョブの UUID
- node_id：実行ノードの UUID
- submitter_id：サブミッターの UUID
- model_id：実行に使用されたモデル
- compliance_zone：ジョブが実行されたコンプライアンスゾーン
- input_tokens：処理された入力トークン数
- output_tokens：生成された出力トークン数
- wall_time_ms：ミリ秒単位の総実行時間
- completed_at：ジョブ完了の RFC 3339 タイムスタンプ
- fee_schedule_id：実行時に適用された BillingFeeConfig の UUID
- input_token_cost_usd：計算された入力トークンコスト（USD、小数点以下 6 桁）
- output_token_cost_usd：計算された出力トークンコスト（USD、小数点以下 6 桁）
- platform_fee_usd：このジョブのプラットフォーム手数料
- node_earnings_usd：このジョブのノードオペレーターの収益
- total_cost_usd：サブミッターへの総コスト

**7.2 署名スキーム**

レシートは Ed25519 を使用して署名されます。ノードはレシートの正規 JSON シリアライゼーション（キーをソートし、空白なし）を登録済み秘密鍵で署名します。署名は base64url エンコードされ、signature フィールドとしてレシートに含まれます。

フェデレーションはノードの登録済み公開鍵を使用してレシートの署名をレシート受信時に検証します。無効な署名を持つレシートは拒否され、ノード整合性アラートをトリガーします。

**7.3 レシートの保存と取得**

フェデレーションはコンプライアンス監査要件をサポートするため、すべてのレシートを最低 7 年間保存します。サブミッターは GET /v1/federation/receipts を通じてレシートを取得できます。ノードオペレーターは GET /v1/federation/nodes/{id}/receipts を通じて自身が実行したジョブのレシートを取得できます。

## 8. 設定可能な経済的精算

MFOP は、請求（署名済みレシートの蓄積）と精算（資金の財務的移動）を分離します。精算は設定可能であり、参加者の種類によって異なるスケジュールで行われる場合があります。

**8.1 BillingFeeConfig**

プラットフォーム管理者は BillingFeeConfig オブジェクトを通じて料金レートを設定します。各 BillingFeeConfig にはバージョン識別子と有効日があり、フェデレーションはジョブ実行時に適用されている設定を使用します。新しい設定はいつでも作成できます。次の請求期間の開始時に有効になります。

BillingFeeConfig フィールド：

- input_token_rate_usd_per_1k：1,000 入力トークンあたりの USD 請求額
- output_token_rate_usd_per_1k：1,000 出力トークンあたりの USD 請求額
- platform_fee_pct：総トークンコストに対するプラットフォームのパーセンテージ（0〜100）
- node_revenue_share_pct：総トークンコストに対するノードオペレーターのパーセンテージ（0〜100、platform_fee_pct との合計が 100 以下でなければならない）
- settlement_period_days：精算の実行頻度（例：30）
- minimum_payout_usd：ノードオペレーターが支払いを受け取るまでの最低蓄積収益額

**8.2 サブミッターへの請求**

サブミッターは後払い方式で請求されます。各精算期間の終了時に、フェデレーションはサブミッターのすべてのレシートを集計し、登録済みの支払い方法に請求します。請求書にはコンプライアンスゾーンおよびモデル別にグループ化されたジョブレシートの明細リストが含まれます。

**8.3 ノードオペレーターへの精算**

ノードオペレーターは、蓄積収益が minimum_payout_usd の閾値を超えていることを条件として、各精算期間の終了時に Stripe Connect 経由で支払いを受けます。閾値を満たさないオペレーターの収益は次の期間に繰り越されます。

## 9. セキュリティモデル

MFOP は、トランスポートセキュリティ、AI 安全ポリシー検証、および実行サンドボックス分離という 3 層のセキュリティモデルを実装します。

**9.1 トランスポートセキュリティ**

すべての MFOP API エンドポイントは TLS 1.3 以上を使用した HTTPS で提供されなければならない (MUST)。プライベートエンタープライズメッシュ展開におけるノードとフェデレーション間の通信には、相互 TLS（mTLS）が推奨される (RECOMMENDED)。API 認証には、X-Channel-API-Key HTTP ヘッダーとして送信される PAK Key を使用します。PAK Key は base64url エンコードされた 256 ビットのランダム値です。

**9.2 AI 安全ポリシー検証**

すべてのジョブの入出力は、実行前およびサブミッターへの配信前に NeMo Guardrails ポリシーに対して検証されます。ベースラインポリシーセット（すべてのコンプライアンスゾーンで必須）には以下が含まれます：

- ジェイルブレーク検出とブロッキング
- 有害コンテンツ検出（暴力、CSAM、自傷行為の助長）
- 出力における PII 漏洩検出
- プロンプトインジェクション検出

特定のコンプライアンスゾーンには追加のポリシーが必要です（付録 B 参照）。

ノードはケイパビリティ広告で指定された NeMo Guardrails ランタイムバージョンを実行しなければならない (MUST)。古い Guardrails バージョンを実行しているノードは DEGRADED としてフラグが立てられ、インストール済みバージョンに存在しないガードレール機能を必要とするコンプライアンスゾーンのルーティングから除外されます。

**9.3 実行サンドボックス分離**

各ジョブは隔離されたサンドボックスで実行されます。ノードは以下のいずれかのメカニズムを使用してサンドボックス分離を実装しなければならない (MUST)：

- gVisor (runsc) — クラウド展開に推奨される (RECOMMENDED)
- Firecracker マイクロ VM — ベアメタル展開に推奨される (RECOMMENDED)
- WASM (Wasmtime) — CPU のみの推論ワークロードに許可される

サンドボックスはジョブ間で破棄・再作成されなければならない (MUST)。持続的なサンドボックス状態（例：モデルの重み）は読み取り専用マウントを通じてジョブ間で共有されてもよい (MAY) が、ジョブ固有の状態（コンテキスト、一時ファイル）はジョブ間で持続してはならない (MUST NOT)。

**9.4 監査ログ**

すべてのジョブルーティング決定、レシート署名、および精算イベントは追記専用の監査ログに記録されます。監査ログは SHA-256 ハッシュを使用して暗号的に連結されます（各エントリには前のエントリのハッシュが含まれます）。監査ログは変更できません。追記操作のみが許可されます。

## 10. ワイヤプロトコル

MFOP はすべての API 通信に HTTPS 上の JSON を使用します。ストリーミングジョブ出力には WebSocket 接続がサポートされています（セクション 10.2 参照）。

**10.1 リクエストおよびレスポンスのフォーマット**

すべてのリクエストおよびレスポンスのボディは UTF-8 エンコードの JSON です。リクエストには Content-Type: application/json を含めなければならない (MUST)。成功レスポンスは HTTP 200 または 201 を使用します。エラーレスポンスは標準エラーエンベロープを使用します：

{ "error": { "code": "<機械可読コード>", "message": "<人間可読メッセージ>", "details": { ... } } }

標準エラーコード：UNAUTHORIZED、FORBIDDEN、NOT_FOUND、VALIDATION_ERROR、QUOTA_EXCEEDED、NO_ELIGIBLE_NODE、COMPLIANCE_VIOLATION、INTERNAL_ERROR。

**10.2 ストリーミング出力**

ストリーミング出力をサポートするノードは wss://{node_endpoint}/v1/jobs/{id}/stream に WebSocket エンドポイントを公開します。クライアントはジョブ提出後に接続します。ノードはトークン出力を JSON フレームのデルタメッセージとしてストリーミングします：

{ "type": "delta", "text": "...", "token_count": N }

ストリームは完了メッセージで終了します：

{ "type": "done", "receipt": { ... } }

完了メッセージ内のレシートはジョブの署名済み BillingReceipt です。

**10.3 冪等性**

ジョブ提出リクエストには Idempotency-Key ヘッダー（UUID）を含めるべきである (SHOULD)。同一の Idempotency-Key を持つリクエストが冪等性ウィンドウ（24 時間）内に受信された場合、フェデレーションはジョブを再実行せずに元のレスポンスを返します。これはネットワークの再試行による重複提出から保護します。
